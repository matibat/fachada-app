# Markdown Pages — Frontmatter Field Reference

This reference documents every field in the `pagesSchema` Zod schema (`src/content/pages.schema.ts`). Use it alongside the [Author's Guide](/resources/markdown-pages) when authoring new pages.

---

## Required Fields

These three fields must appear in every page file or the build will fail with a Zod validation error.

### `title`

```yaml
title: "My Page Title"
```

- **Type**: `string`
- **Used for**: `<title>` tag, `og:title`, page heading in link cards
- **Guideline**: Keep under 60 characters. Include the app name suffix for discoverability (e.g., `"Asset Manual — Fachada"`).

---

### `description`

```yaml
description: "A concise 1–2 sentence summary of this page's content."
```

- **Type**: `string`
- **Used for**: `<meta name="description">`, `og:description`, card descriptions in hub pages
- **Guideline**: Aim for 150–160 characters. This text appears in search engine results and social link previews.

---

### `apps`

```yaml
# Target a single app:
apps: ["default-fachada"]

# Target multiple apps:
apps: ["default-fachada", "artist-engineer"]

# Make available to all apps:
apps: "*"
```

- **Type**: `string[]` or `"*"`
- **Used for**: Filtering pages at build time so each app only renders its own content
- **Guideline**: Always use the array form for app-specific content. Only use `"*"` for truly shared cross-app documentation.

---

## Optional Fields

All of the following fields are optional. Omitting them does not cause the build to fail, but including them unlocks additional features.

---

### `path`

```yaml
path: "/resources/markdown-pages"
```

- **Type**: `string`
- **Used for**: Determining the URL route for this page
- **Default if omitted**: `/pages/{filename-without-extension}`
- **Guideline**: Use explicit paths when the URL must fit within an existing resource hierarchy (e.g., `/resources/...`). Paths must be unique across the app — duplicates cause a runtime warning and the second page will be skipped.

---

### `keywords`

```yaml
keywords:
  - markdown pages
  - content authoring
  - frontmatter
```

- **Type**: `string[]`
- **Used for**: `<meta name="keywords">`, `llm.txt` enrichment, search indexing signals
- **Guideline**: List 3–8 specific terms. Avoid generic filler words. These are surfaced in the AI-readable `llm.txt` endpoint to help language models understand page topics.

---

### `llmSummary`

```yaml
llmSummary: "Step-by-step guide to authoring Markdown pages in Fachada. Covers frontmatter, app targeting, and cross-linking."
```

- **Type**: `string`
- **Used for**: The `/llm.txt` file generated by `LlmTextGenerator`, which provides a machine-readable summary of the site for AI crawlers
- **Guideline**: Write 1–2 sentences in plain prose. Avoid markdown syntax — this text is consumed programmatically. If omitted, the `description` field is used as a fallback.

---

### `ogImage`

```yaml
ogImage: "/images/og-my-page.png"
```

- **Type**: `string`
- **Used for**: `og:image` meta tag for social sharing previews
- **Guideline**: Should be an absolute path beginning with `/` pointing to an image in `public/`. Recommended dimensions: 1200×630px. If omitted, the app-level `ogImage` from `siteConfig` is used as a fallback.

---

### `downloadFilename`

```yaml
downloadFilename: "my-page.md"
```

- **Type**: `string`
- **Used for**: Renders a `DownloadButton` component on the page that allows readers to download the raw Markdown source
- **Guideline**: Set to a descriptive kebab-case filename with `.md` extension. The template serves the Markdown source of the current page as the download payload, using this string as the suggested filename in the browser's download dialog. Omit if the page is not intended to be downloaded (e.g., interactive or dynamic content).

---

### `backLink`

```yaml
backLink:
  href: "/resources/markdown-pages"
  label: "Author's Guide"
```

- **Type**: `{ href: string; label: string }`
- **Used for**: Renders a "← Back" navigation link at the top or bottom of the page
- **Guideline**: Use `href` for the absolute path of the preceding page in a reading sequence. Use `label` for the visible link text (typically the short title of the previous page). Pair with `nextLink` on adjacent pages to create a linear navigation flow.

---

### `nextLink`

```yaml
nextLink:
  href: "/resources/markdown-pages-reference"
  label: "Frontmatter Field Reference"
```

- **Type**: `{ href: string; label: string }`
- **Used for**: Renders a "Next →" navigation link at the bottom of the page
- **Guideline**: Mirrors `backLink` but points forward in the reading sequence. The `label` should be the short title of the next page, not a generic "Next" or "Continue" string, so users know what they are navigating to before clicking.

---

## Full Example

The following frontmatter block demonstrates every field in a single page:

```yaml
---
title: "Complete Example Page — Fachada"
description: "A demonstration page that uses every supported frontmatter field in the Fachada pages schema."
apps: ["default-fachada"]
path: "/resources/complete-example"
keywords:
  - example
  - frontmatter
  - fachada pages
llmSummary: "Demonstration of all Fachada page frontmatter fields, including optional SEO, navigation, and download fields."
ogImage: "/images/og-complete-example.png"
downloadFilename: "complete-example.md"
backLink:
  href: "/resources/markdown-pages"
  label: "Author's Guide"
nextLink:
  href: "/resources/markdown-pages-reference"
  label: "Field Reference"
---
```

Use this as a template when creating a new page that needs maximum discoverability and navigation integration.

## Schema Source

The canonical schema lives in:

```
src/content/pages.schema.ts
```

If a field is added to the schema in the future, it will appear in that file first. This reference document is updated alongside each schema change.
